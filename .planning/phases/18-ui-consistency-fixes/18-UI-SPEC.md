## FIX-01: Toast Interaction Contract

### Problem (current state)

After the user submits the add-item form, the server redirects to `/?toast=Item+added%2C+scanning+in+background`. The `?toast=` query param is read on `DOMContentLoaded` and fed to `window.showToast()`. However a separate scan panel UI (`#scanPanel`, `.scan-pill`, `.scan-card`) also activates — creating a second distinct boxed UI running simultaneously.

The requirement is that the post-add "scanning in background" message uses **only** the `#toast` primitive and does not spawn or co-activate the floating scan panel box.

### Required behavior

1. After item add, `#toast` shows "Item added — scanning in background" for 4000ms then auto-dismisses.
2. The `#scanPanel` floating pill does NOT auto-activate on post-add navigation (unless a scan was already running before the add).
3. `window.showToast(message, durationMs)` is the sole mechanism. No secondary `.scan-card` or `.scan-pill` shown for the add-confirmation step.
4. If a scan does start in the background (triggered server-side after add), the scan panel activates only once the polling detects `is_running: true` — not from the redirect itself.

### Toast visual spec

The `#toast` element already has the correct styling from `static/style.css` section 14:

```
position: fixed; bottom: 80px; right: var(--space-lg); z-index: 55;
background: var(--color-surface); border: 1px solid var(--color-border);
border-radius: 0; padding: var(--space-sm) var(--space-lg);
box-shadow: 0 8px 24px rgba(0,0,0,0.4);
```

No CSS changes required for FIX-01. Only JavaScript/template wiring changes.

### Toast copy (FIX-01)

| Trigger | Copy | Duration |
|---------|------|----------|
| Item successfully added | "Item added — scanning in background" | 4000ms |
| Item add failed (server error) | "Something went wrong. Try again." | 4000ms |

Use em-dash (—) not hyphen (-) for the separator, consistent with existing scan toast copy in `renderStatus()`.

## FIX-02: Placeholder Image Contract

### Problem (current state)

`templates/item_detail.html` uses `vinyl-placeholder.svg` (658 bytes, minimal SVG) as both the primary placeholder (no artwork) and the `onerror` fallback. The new `empty-vinyl-placeholder.png` asset (780 KB) exists in `/static/` but is not yet wired.

### Required behavior

1. When `item.artwork_url` is null/empty: render `<img src="/static/empty-vinyl-placeholder.png">`.
2. When `item.artwork_url` is present but fails to load (onerror): fall back to `/static/empty-vinyl-placeholder.png`.
3. Same 120x120px dimensions, `object-fit: cover`, `border-radius: 0` as current.
4. `alt` text unchanged: `"{{ item.query }} — no artwork available"` for the no-artwork case.
5. Dashboard card artwork fallback (`onerror` on `.card-artwork` elements) is out of scope for this phase — only item detail.

### Image spec

| Property | Value |
|----------|-------|
| Asset | `/static/empty-vinyl-placeholder.png` |
| Dimensions | 120x120px rendered (intrinsic size may differ — `object-fit: cover` handles it) |
| Border radius | 0 |
| Loading | `loading="lazy"` (below-fold on most viewports) |

## Interaction States

### Toast (#toast)

| State | Trigger | Visual |
|-------|---------|--------|
| Hidden | Default | `display: none` (`.hidden` class) |
| Visible | `showToast()` called | `display: flex`, visible for `durationMs`, then auto-hides |
| Replaced | New `showToast()` call while visible | Previous timer cleared, new message shown, timer reset |
