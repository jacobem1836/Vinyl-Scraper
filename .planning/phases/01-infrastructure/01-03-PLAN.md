# Plan 3: Frontend Polling UX — Scanning Indicator + Dashboard Auto-Refresh

**Phase:** 1 — Infrastructure
**Goal:** After adding an item, the dashboard shows a "scanning" indicator on the new card; the card auto-refreshes when listings appear, without requiring a manual page reload
**Requires:** Plan 1 (scan decoupling must be in place — items now appear on dashboard before scan completes)
**Requirements:** PERF-01 (UX side of the decoupling)

## Tasks

### Task 1: Add a scanning status API endpoint

**File(s):** `app/routers/wishlist.py`
**What:** Add a lightweight API endpoint that returns the scan status of a single item — used by the frontend to poll whether listings have appeared yet.
**How:**
1. Add a new route `GET /api/wishlist/{item_id}/status` (no API key required — it's used by the dashboard JS, not the iOS Shortcut):
   ```python
   @web_router.get("/wishlist/{item_id}/status")
   async def item_scan_status(item_id: int, db: Session = Depends(get_db)):
       item = db.query(WishlistItem).filter_by(id=item_id, is_active=True).first()
       if not item:
           raise HTTPException(status_code=404, detail="Item not found")
       listing_count = db.query(Listing).filter_by(
           wishlist_item_id=item_id, is_active=True
       ).count()
       return {
           "id": item_id,
           "has_listings": listing_count > 0,
           "listing_count": listing_count,
           "last_scanned_at": item.last_scanned_at.isoformat() if item.last_scanned_at else None,
       }
   ```
2. This endpoint is deliberately lightweight — a count query, not a full enrichment. It's called every few seconds by JS polling.

**Acceptance:** `GET /wishlist/1/status` returns JSON with `has_listings`, `listing_count`, and `last_scanned_at`. Response time < 100ms.

### Task 2: Add scanning spinner and polling JS to dashboard template

**File(s):** `templates/index.html`, `static/style.css` (or equivalent CSS file)
**What:** When an item has no listings and was just added (indicated by a `scanning=true` query param or by `listing_count == 0` and `last_scanned_at is None`), show a subtle spinner on the card. JavaScript polls the status endpoint and refreshes the card when listings appear.
**How:**
1. **Template changes (`templates/index.html`):**
   - In the item card rendering loop, add a conditional block: if `item.listing_count == 0` and `item.last_scanned_at is None`, add a CSS class `card--scanning` and a small spinner element (CSS-only spinner, no image asset) (per D-05)
   - Add `data-item-id="{{ item.id }}"` attribute to each card div for JS targeting
   - Add `data-scanning="true"` attribute on cards that are in scanning state

2. **CSS changes:**
   - Add `.card--scanning` styles: a subtle pulsing border or a small spinner icon in the corner
   - Spinner: pure CSS using `@keyframes spin` on a small `border` element (no external assets)
   - Keep it subtle — per the CONTEXT.md specifics: "not intrusive"

3. **JavaScript (inline in template or in a small `static/poll.js` file):**
   - On page load, find all elements with `data-scanning="true"`
   - For each, start a `setInterval` polling loop (per D-04):
     - `fetch('/wishlist/{id}/status')` every 5 seconds
     - When `has_listings` is `true`, reload the page (simple approach) or replace the card content via a targeted fetch
     - Stop polling after 2 minutes (timeout — scan should have completed; avoids infinite polling)
   - Use simple `fetch()` — no library needed
   - Total JS: ~20-30 lines

4. **Pass scanning state from backend:**
   - In `app/main.py` `index` route: the enriched items already include `listing_count` and `last_scanned_at`. No backend change needed — the template can derive scanning state from `item.listing_count == 0 and item.last_scanned_at is None`.

**Acceptance:**
- Adding a new item via the web form shows the item on the dashboard immediately with a spinner
- After 5-15 seconds (background scan completes), the page auto-refreshes and listings appear on the card
- The spinner disappears once listings are populated
- Polling stops after 2 minutes even if scan never completes (no infinite network requests)
- Items that already have listings show no spinner and trigger no polling

## Verification

1. Start the app: `uvicorn app.main:app --reload`
