# Milestone v1.2: Signal Intelligence & Notifications

**Status:** ✅ SHIPPED 2026-04-14
**Phases:** 13–15
**Total Plans:** 8

## Overview

Added signal filtering (relevance scoring, ships_from enrichment, digital listing suppression), consistent UI feedback primitives (toast system, modal defaults), and a full notification system (back-in-stock alerts, price-drop alerts, cooldown deduplication, digest email aggregation).

## Phases

### Phase 13: Signal Filters

**Goal:** Filter listing noise so only relevant, buyable vinyl reaches the dashboard — suppress digital-only results, enrich with ships_from data, and rank by relevance.
**Depends on:** Phase 12 (UI Fixes Round 2 complete)
**Plans:** 3 plans

Plans:
- [x] 13-01: Relevance scoring helper + digital listing filter module
- [x] 13-02: ships_from enrichment via Discogs marketplace API
- [x] 13-03: Wire digital filter + relevance scoring + filter log into scanner and dashboard/detail routes

**Details:**
- Relevance scoring helper (`app/services/relevance.py`) scores listings against item query
- Digital listing filter (`app/services/digital_filter.py`) suppresses format-code-only items
- `ships_from` populated from Discogs marketplace API on every scan
- Relevance threshold applied at query time in dashboard + detail routes

### Phase 14: Feedback Primitives UI Hint

**Goal:** Make every user-initiated action emit consistent feedback through the single `#toast` primitive, and guarantee the add-item modal always opens with Type = "album".
**Depends on:** Phase 13 complete
**Plans:** 1 plan

Plans:
- [x] 14-01: Add window.showToast() helper, wire scan completion to toast, reset modal type on every open

**Details:**
- `window.showToast()` helper added to base.html — single toast primitive for all feedback
- Scan completion wired into `renderStatus()` via polling — toast on done with listing count
- Add-item modal type selector reset to "album" on every open via capture-phase click listener
- Server-side `?toast=` redirect chain feeds the same #toast element

### Phase 15: Notification Expansion

**Goal:** Users receive deduped back-in-stock + price-drop alerts as unified digest emails.
**Depends on:** Phase 14 complete
**Plans:** 4 plans

Plans:
- [x] 15-01: Schema foundation — `prev_price`, `prev_is_in_stock`, `last_notified_at`, `notify_drop_mode`, `notify_drop_pct`, `notify_drop_usd` columns + config defaults
- [x] 15-02: Notifier unit tests (failing) — TDD for detection helpers
- [x] 15-03: Detection helpers (`_price_dropped`, `_back_in_stock`) + digest template (`templates/digest_alert.html`)
- [x] 15-04: Scheduler collect-then-dispatch pattern — `_collect_events` helper + two-phase scan/digest

**Details:**
- `_price_dropped()` detects % or USD price drops vs prev_price snapshot
- `_back_in_stock()` detects listing transitioning from out-of-stock to in-stock
- `_within_cooldown()` suppresses repeat sends within configurable hours window
- `send_digest_email()` renders all events in a single CRATE-styled digest email
- Scheduler refactored to collect-then-dispatch: scan all items, collect events, send one digest
- `last_notified_at` committed to DB on successful send
- 9 unit tests covering NOTIF-01 through NOTIF-04

## Milestone Summary

**Key Decisions:**

- Relevance scoring uses query term overlap — no external ML dependency
- `ships_from` sourced from Discogs marketplace listings API (not scraped HTML)
- Single `#toast` primitive for all UI feedback — no secondary dialog elements
- `notify_drop_mode` is per-item (`pct` or `usd`) — user controls threshold type
- Cooldown window is global config (`notify_cooldown_hours`) not per-listing
- Collect-then-dispatch pattern: one digest per scheduled scan run, not per-listing
